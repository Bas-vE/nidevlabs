Remarks

Before implementing the NIPath, UNC path is serialized as separate components, such as @"\2poseidon\incoming\tlan" was serialized as "\2poseidon"incomingtlan. I used to follow this form. However, LabVIEW Runtime parses UNC path such that the first two components (the machine and share folder) of path data are considered as one component. What this requires us to do on our end, is ensure that when we parse an NIPath describing a UNC path, that we combine the machine and share as a single component. To deal with current persisted UNC paths, we should make our source model parser deal with both the current persisted forms, as well as the new form where the machine name and share are combined.
