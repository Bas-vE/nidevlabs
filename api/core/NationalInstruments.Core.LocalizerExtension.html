Remarks

Locating a localized string resource involves the following elements: a) A resource key b) A class containing the resources c) An assembly containing the resources

The extension tries to offer flexibility and leniency when locating a string resource. To locate the string, the localizer uses the following approach: 1) Try the Key property if not null or empty. 2) If Key is null or empty, or we fail using Key, try the target object's x:Name (when using 'abbreviated' syntax). 3) If all of these fail, a general (localized) error string is returned.

The implementation uses a ResourceManager to retrieve the resources, which requires a class name and the assembly containing the resources class.

How we select a ResourceManager: 1) Via the ResourceClassName + AssemblyName properties If either / both are defined, and we locate the resource, use the properties. If one of these properties is undefined (null), a series of default fallbacks is triggered. See further remarks below. 2) Default Mechanism If (1) does not apply, or fails, then attempt to introspect using the IServiceProvider argument to the ProvideValue() function. This mechanism locates the 'root' object in the FrameworkElement's ownership chain (if the target object is a FramworkElement) and uses that class's name as the basis to locate a resource manager. 3) If we fail to locate a resource manager in (1), (2) or (3), use this library's resources.

To determine which assembly to use, the extension uses the following decision process: 1) Attempt using the value of AssemblyName if not null or empty. 2) If (1) fails, try using the assembly containing the type of the target object. If that fails, and the target object is a FrameworkElement, move up the parent chain, attempting to locate a parent that is in the same namespace as ResourcesClassName (if ResourcesClassName is not null or empty). If the narrowest namespace check fails, repeat the using the next broader namespace. For example, if ResourcesClassName is NationalInstruments.Shell.Dialogs.SharedResources but 'AssemblyName' is null, first check the parent hierarchy looking for types in NationalInstruments.Shell.Dialogs, then NationalInstruments.Shell, and finally, NationalInstruments. If the namespace criteria is met, then the assembly containing that type will be used. 3) If (1) and (2) fail, use the assembly containing the type described by IRootObjectProvider, which is retrieved from IServiceProvider. 4) If all other attempts to locate an assembly fail, use the assembly containing this type.

When using the ResourceChain, the extension will search for the resource string applying the above rules, potentially examining multiple locations for the string resource.
