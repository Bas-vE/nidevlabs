ModelSideEffects

Helpers to apply a "side effect" transaction. A side effect transaction is a modification to a source model in response to a modification of another source file. This is hard because of the locking rules where you cannot acquire a write lock while holding another lock. These helpers deal with this rule by implementing an asynchronous inspection and application strategy. Basically the time line looks like this:

Receive model change notification Schedule side effect

  • await for changed model to be unlocked
  • acquire upgradeable read lock on target model
  • acquire read lock on changed model Inspect - Both the modified and target models are read locked and can be accessed
  • release read lock on changed model
  • upgrade to write lock
  • begin transaction Apply changes to target model based only on inspection results
  • commit transaction
  • unlock everything

These helpers will handle transacting in the correct thread There are several patterns of inspection and apply changes that are all lock and thread correct

TransactionSystem

A TransactionSystem is a DefinitionBoundary that owns all the necessary components for transacting changes to the model. These pieces supply support for undo/redo, property change notification, change validation through ModelBatchRules, and locking services. TransactionSystems generally exist along file boundaries. This means that it is common for each file to have its own transaction system. This means that generally each file has its own undo/redo stack, its own notion of whether it is dirty, its own collection of ModelBatchRules to ensure state, and it's own locking services.

The locking services provided by a transaction system are hierarchical in nature. For instance, when a read lock is acquired (through AcquireReadLock()), a read lock is first acquired on the root most transaction system, and then down the descendant chain until we reach the transaction system on which the request was originally made.
