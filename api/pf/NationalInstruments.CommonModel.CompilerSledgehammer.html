EnableToggleName

Name for the temporary toggle that resurrects "The Sledgehammer" recompile strategy (https://nitalk.jiveon.com/docs/DOC-438421). Potentially still useful to triage compiler gremlins as we harden the Cardassia B release.

USE WITH EXTREME CARE. Enabling this toggle can corrupt your caches! The old workflow is also no longer actively maintained. Recommended MINIMUM precautions if you enable:

  1. Only change the toggle while LabVIEW is not open (i.e. Edit Preferences.XML to add the boolean property).
  2. Delete existing .cache folder BEFORE loading any project whenever you have the toggle enabled.
  3. Delete .cache folder AFTER testing with the toggle enabled. NEVER save/share caches generated with the sledgehammer enabled. This is because the cache contents won't be compatible with the new default compile flow, even though there's no version identifier difference to invalidate the cache for normal builds.

Remarks

Note the crucial '!' operator; this important because default feature toggle state has flipped since sledgehammer refactoring started. Historically, the default was "sledgehammer enabled", but now we want the default to be NOT ENABLED. So we need a feature toggle flag with default value false, but there's lots of code that uses the old boolean polarity.

Also, this property intentionally uses GetPreference directly rather than IsIncompleteFeatureEnabled. This is because "incomplete features" are always filtered out on distribution (installed) builds, but we want the flexibility to test installed builds with the old sledgehammer behavior.

TL;DR: Coding the feature toggle this way gets the correct default without broader impact on sensitive code.
