AttachAnonymousPostTransactionSideEffect(Object, Action)

Attach an anonymous post transaction side effect which will get attached to the current active Transaction and execute when
the transaction is undone. If the same side effect instance is registered more than once for the same object, it will only get executed once.

Declaration
void AttachAnonymousPostTransactionSideEffect(object registeringObject, Action sideEffect)
Parameters
Type Name Description
Object registeringObject

The object registering this side effect. This is typically the object the side effect will act upon in some way.
Action sideEffect

The side effect action to reapply when the transaction is undone.
Remarks

This function behaves the same as AttachNamedPostTransactionSideEffect(Object, String, Action) except that anonymous side effects always remain in effect until the transaction it's attached to is discarded from the undo stack and can't be removed manually. If you need that capability, you should create a named side effect instead. Named side effects are encouraged over anonymous side effects since they provide better control over the lifetime of the side effect and help avoid redoing work that might be overwritten by a subsequent anonymous side effect.

AttachNamedPostTransactionSideEffect(Object, String, Action)

Attach a named post transaction side effect which will get attached to the current active Transaction and execute when the transaction is undone. A named side effect will replace a side effect already registered with the same name for the same object.

Declaration
void AttachNamedPostTransactionSideEffect(object registeringObject, string sideEffectName, Action sideEffect)
Parameters
Type Name Description
Object registeringObject

The object registering this side effect. This is typically the object the side effect will act upon in some way.
String sideEffectName

The name the side effect is registered with. If the same name is registered more than once for the same , the previously registered side effect will be replaced with the new provided to this call.
Action sideEffect

The side effect action to reapply when the transaction is undone.
Remarks

Side effects are typically added when a change from another transaction system is driving a side effect change to this transaction manager, and that change needs to remain in effect even across an undo. If effect, the side effect makes the change such that it isn't undoable.

Once a side effect is executed from undo, the side effect will get reattached to a new non-user transaction at the end of the undo stack. The side effect will remain there and get executed again only if the non-user transaction is undone. This will happen if either a redo is issued where by all non-user transactions at the end of the stack are rolled back before the user transactions on the redo stack can be recommitted, or another undo is issued causing the non-user transaction to get rolled back. In either case, the side effect will get reattached to a non-user transaction at the end of the undo stack, and the process repeats. The side effect will remain in effect until either removed by calling RemoveNamedPostTransactionSideEffect(Object, String) or the transaction the side effect is attached to is discarded from the undo stack.

ActivityAwaitable<IAsyncActiveTransaction>

ActivityAwaitable<T>; await it to get the new active composite transaction. Transaction body will execute on appropriate thread as described above.
Remarks

This method automatically implements background-aware transactions based on which thread calls it and the state of this transaction manager. This method waits until the transaction system is idle so that the new transaction is always the outer composite transaction.

Transaction body may execute on different threads depending on conditions:

  1. Awaiter will continue on UI thread if ANY of these conditions are true: 1A. Model does not allow background edits 1B. Caller is on the UI thread

  2. Awaiter will continue on background activity thread if ALL of these conditions are true: 2A. Model allows background edits 2B. Caller is on a background thread

REMINDER: Always await CommitAsync() on returned transaction! Failure to do so could cause the rest of your code in the calling method to execute on an unexpected thread. This includes cases where the transaction body throws an exception, which will skip a commit call inside the using block. Recommended best practice: encapsulate your using block with Begin and Commit calls in a private async method that contains no other code after commit. The await at the caller level should return your code to that original threading context.
