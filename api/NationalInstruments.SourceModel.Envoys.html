BindOnTargetedAttribute

Declares that service should be bound to a project item immediately when the item is under a real target, or on demand (when a provided service is queried for). Specifying ImmediateBind is the most expensive binding mode since it will bind to all matching envoys at load, even for Envoys which are unused in the NULL target. This is preferred over ImmediateBind since it will not bind to objects under the NULL target, but will immediately bind to all targeted items.

BindsToModelDefinitionTypeAttribute

Attribute which declares which project item "ModelDefinitionType" that the service supports. The service will only be bound to project items which have a matching model definition type.

This attribute can be used more than once to specify that the service binds to multiple model definition types.

If a service does not provide any ModelDefinitionTypes then it is assumed that the service is model definition type agnostic.

When registering a binding to a "ModelDefinitionType", it's recommened that you also provide a namespace name for the type. This namespace name should be the same name as provided by an INamespaceSchema and allows for the possibility of running future mutations on this type.

SparseEnvoyManager

A sparse envoy manager refers to a real EnvoyManager via a NationalInstruments.SourceModel.Envoys.SparseEnvoyManager.SparseEnvoyManagerDependency. Envoys are copied from the real envoy manager to the sparse envoy manager as necessary to resolve dependencies. There are multiple reasons to use a SparseEnvoyManager as opposed to adding linked SourceFileRefs to each target:

  • You care about the sparseness. ie: You don't want to duplicate all envoys from the EnvoyManager into the targets.
  • Your Definition/EnvoyManager cannot handle multi-context editing.
  • You want different user settings for envoys under different targets. ex: A VI might cache different info depending on the target. In that case, you would want a SparseEnvoyManager so that there can be different cached user settings. The real envoy manager can be an EnvoyManagerFile or a Project both of which are actually EnvoyManagers. If the envoy manager is the project, the AssociatedRootEnvoy is actually an envoy below the project used to limit the scope of the sparse envoy manager.

DateTimeSignature

Class that wraps a System.DateTime. The purpose of this class is to centralize the comparison of timestamps. Because there is no exposed method to access the System.DateTime.Ticks of the wrapped System.DateTime, this prevents clients from incorrectly comparing timestamps with exact precision. The accuracy of timestamps varies in multiple ways. E.g. when a file is created on a FAT file system, the timestamp is rounded up to the nearest even second. When a file is packaged and installed by nipm, the timestamp is cut off at seconds. Because of this, it is not safe / correct to compare them exactly. The Equals method of this class considers timestamps that are within 2 seconds of each other to be equal. This has been determined to be correct for all scenarios we are aware of.

EnvoyAndLoadResult

A custom result returned from APIs that synchronously add an Envoy to a project, and then immediately begin the load of that Envoy. This result contains the Envoy that was added synchronously, and a Task representing the load of that Envoy. This enables callers of APIs that do this type of work to transact around the add of the Envoy, but not around the load of the Envoy.
