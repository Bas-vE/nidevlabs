Remarks

This method starts a transaction to update the OverridingModelDefinitionType for the specific envoy being converted and discards the undo/redo stack. However, this will not explicitly save the owning envoy manager (project) to persist the new OMDT. There's no guarantee that all the linked envoys will have their OMDT updated. If the OMDT is missing, it will be detected and re-added when the envoy is loaded. That will be persisted when the owning envoy manager (project) is saved. It is theoretically possible that the missing OMDT could cause problems, but the risk is low. This is a concept that other document types have already been dealing with. An example problem would be a envoy service that immediately binds to the protected OMDT or one that only binds to VIs that don't have the protected OMDT.
